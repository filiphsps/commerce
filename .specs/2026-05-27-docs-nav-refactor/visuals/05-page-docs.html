Multi-tenancy

One deploy serves many shops. Tenant resolution happens at the edge by hostname, then every downstream call carries explicit { shop, locale }. The App Router never sees an un-tenanted request.

Nordcom Commerce is multi-tenant by hostname. There is no per-shop deployment — adding a new shop is a row in the shops MongoDB collection, no redeploy. The middleware reads the request's hostname, resolves the shop, and rewrites the URL into /[domain]/[locale]/… before the App Router takes over.

The resolution chain

Request lifecycle · hostname → tenant-scoped route
1 Inbound request hits the edge with raw Host: header Could be primary, alternative, or dev-alias
↓
2 Middleware looks up Hostname → Shop in MongoDB via resolveShopFromHostname
↓
3 Locale resolved via fallback chain request → shop default → platform default
↓
4 URL rewrites to /[domain]/[locale]/… App Router sees the tenanted path
↓
5 Every data call carries { shop, locale } explicitly no implicit tenant context, ever
Concept
Tenant context is never implicit. There's no getCurrentShop() singleton, no request-scoped global, no AsyncLocalStorage tenant slot. Every Shopify call constructs a fresh ShopifyApolloApiClient({ shop, locale }); every CMS reader takes { shop, locale } in its arguments object. New data-fetching helpers must follow this contract.

Data fetching contract

Every reader in @nordcom/commerce-cms/api follows the same shape: tenant-scope on shop.id, apply locale, fall back to shop default, return null when the entity doesn't resolve. Throws NotFoundError when the slug is structurally invalid.

apps/storefront/app/[domain]/[locale]/blog/[slug]/page.tsx 
import { getArticle } from '@nordcom/commerce-cms/api';

export default async function ArticlePage({ params }: {
  params: Promise<{ shop: ShopDomain; locale: LocaleCode; slug: string }>;
}) {
  const { shop, locale, slug } = await params;  // async params in Next 16
  const article = await getArticle({ shop, locale, slug });

  if (!article) notFound();
  return <ArticleView article={article} />;
}
Watch out
Never derive the shop from a singleton or environment variable. The deployment serves many shops simultaneously; a wrongly-cached tenant context would leak data across shops. If you find yourself reaching for a global, stop and pass { shop, locale } through instead.

The tenant-scoped cache invariant

Every cached entity's tags include a shop key. Revalidating one tenant cannot touch another's data. Inside a tenant-scoped cache, entities are further keyed by locale — locale is a qualifier under tenant, not above it.

cache key hierarchy 
cms.<tenantId>.<collection>.<key>   # individual entity
cms.<tenantId>.<collection>         # collection sweep
cms.<tenantId>                      # whole-tenant sweep
Tip
For helpers that build cache tags, use cmsTenantRootTags(shop) from @nordcom/commerce-cms/cache. Never hand-format these strings — the invariant is enforced by the helper.
