Surface 07 · /search

Search

Search crashes today and has no filtering. The fix is two-fold: guard every state so a missing query, a malformed product, or a zero-result query can never throw — and reuse the shared faceted filter (the same component as /products and collections) plus the one true product card/grid, so search stops being a special case.

1 The crash

Search results are fetched in search-content-gate.tsx from cachedSearch (api/shopify/search.ts) and rendered through search-content.tsx. Two unguarded paths converge: search edge nodes are unsafe_cast<Product> with no runtime guarantee of the fields the card reads, and the client has no landing state for the no-query case — so the page either throws on the first malformed product or paints blank.

Before · throws / blank
GET /search?q=slides
→ cachedSearch() → SearchApi
→ node = unsafe_cast<Product>(edge.node)
→ ProductCard reads node.variants.edges[0]
→ TypeError: cannot read 'edges' of undefined

GET /search (no q)
→ products: [] → <section class="empty:hidden">
→ collapses to nothing (no landing)
After · always a guarded state
q present + results → grid + shared filter
q present + zero hits → branded .empty
no q → search landing (popular + trending)

malformed node → skipped, never deref'd
→ no path throws, no blank page
!
CauseIn api/shopify/search.ts the search edges are mapped with unsafe_cast<Product>(item.node) — the comment claims "the Storefront API guarantees all queried fields are present", but PRODUCT_FRAGMENT_MINIMAL has historically shipped without the variants the card dereferences. An unguarded data.variants.edges[0] in the product card then throws for any node missing the field. Compounding it, search-content.tsx only renders a branded empty state when activeQuery.length > 0; with no query it falls through to <section class="… empty:hidden">, which collapses to a blank page instead of a real no-query landing. Filtering is rendered as a loose <Filters> card row, not the shared faceted aside — so search "lacks filters" in the consolidated sense.
✓
Fix (design)Three explicitly guarded states, none of which can throw: (1) results — query + hits → shared .shop-layout faceted aside + .pc-grid; (2) no results — query + zero hits → the shared .empty with suggestions + "Browse all products"; (3) no query — empty q → a search landing (big field, popular searches, trending grid). Product nodes are normalized/guarded before render so a missing variants field skips the card rather than dereferencing undefined.

DRYEvery interactive piece here is shared, not rebuilt: the faceted aside is the SAME .filters / .facet / .toolbar component as /products and collections; the cards are the one true .pc / .pc-grid; the zero-hit block is the shared .empty; the chips are .filter-chip. Search owns only the field row and the no-query landing copy.
i
URL contractFilters reuse the same query params as the rest of the catalog: ?q=slides&vendor=acme-studio&color=sand&size=42&price=25-50&sort=price-asc&availability=in-stock. The shared filter reads and writes these params, so a filtered search is a shareable, indexable URL — and the result count + active-filter chips derive from them.
