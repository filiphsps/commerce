Admin · Customization

Store-wide default configuration

One place to set how every storefront block and component behaves by default — product-card button placement, section visibility, block options — with a clean cascade so any default can be overridden per surface or per block instance. Built to extend to future custom, user-provided blocks without new plumbing.

Locked decisions
1
Architecture home
Realize the existing ShopExtensionManifest as the storage + cascade home and wire it into the storefront render path. Unify the admin UI: theme tokens and component/block defaults render through one shared editor shell, extracted from today's bespoke theme editor.
2
Override model
Four-level cascade — platform → store → surface → per-instance — surfaced as an explicit Inherit ↔ Override control per setting, reusing the locale-bucket / responsive "base + override" mental model. Empty = inherit.
3
Settings contract
Blocks declare settings: FieldDescriptor[] alongside their content fields; non-block components (product card, header chrome) register through a small component registry. Both feed the same editor. Custom blocks inherit settings support via their descriptor.
4
Information architecture
A unified Customization hub with tabs — Theme · Components · Blocks · Sections. The existing theme catalog becomes the Theme tab; search spans the hub. /settings/theme redirects in.
The cascade

Resolution is per-setting and last-write-wins down the chain. A tier only contributes the keys it explicitly sets; everything else falls through. Tier colors below are reused as the source chips throughout the editor.
